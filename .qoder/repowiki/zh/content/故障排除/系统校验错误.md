## 验证脚本功能解析

### simple-validate.sh 核心功能
该脚本执行轻量级系统完整性检查，主要包含以下五个方面：

1. **项目结构验证**：确认 `backend/src`、`frontend/src`、`config`、`knowledge` 等关键目录存在。
2. **核心文件检查**：验证后端服务（如 `app.js`、`LLMService.js`）和前端组件（如 `App.tsx`、`HomePage.tsx`）是否存在。
3. **配置文件验证**：确保 `config/default.json`、`development.json`、`production.json` 存在且可读。
4. **知识库内容扫描**：使用 `find knowledge -name "*.md"` 统计 `.md` 文档数量。
5. **测试覆盖率统计**：分别统计前后端测试文件数量。

此脚本适用于快速部署前的基础检查。

**Section sources**
- [simple-validate.sh](file://simple-validate.sh#L1-L120)

### validate-system.sh 深度校验机制
相比简化版脚本，`validate-system.sh` 提供更全面的功能验证，涵盖九个维度：

1. **项目结构完整性**
2. **关键文件存在性**
3. **配置文件格式正确性（JSON语法）**
4. **知识库内容充足性**
5. **测试文件完整性**
6. **代码质量（语法检查）**
7. **依赖包配置合理性**
8. **文档完整性**
9. **功能模块覆盖度**

此外还进行系统集成验证，包括 API 路由定义、中间件集成、前后端接口对接等。

特别地，该脚本会对 `config/*.json` 文件进行 JSON 格式解析测试，并调用 Node.js 执行语法检查，防止因配置错误导致启动失败。

## 常见错误类型与修复方案

### LLM_CONFIG_MISSING 错误
#### 含义说明
表示大模型配置文件 `configs/llm-config.json` 缺失或路径不正确。该文件用于指定活跃的 LLM 提供商（如 Ollama、OpenAI）、API 地址、模型名称及参数。

#### 修复命令
```bash
# 创建默认配置文件
mkdir -p configs
cat > configs/llm-config.json << 'EOF'
{
  "version": "1.0.0",
  "active_provider": "mock",
  "providers": {
    "ollama": {
      "name": "Ollama",
      "type": "local",
      "endpoint": "http://localhost:11434",
      "models": {
        "primary": "llama2",
        "fallback": "qwen2"
      },
      "parameters": {
        "temperature": 0.7,
        "max_tokens": 2048,
        "timeout": 30000
      },
      "enabled": true
    },
    "openai": {
      "name": "OpenAI",
      "type": "remote",
      "endpoint": "https://api.openai.com/v1",
      "api_key": "${OPENAI_API_KEY}",
      "models": {
        "primary": "gpt-3.5-turbo"
      },
      "enabled": false
    },
    "mock": {
      "name": "Mock Provider",
      "type": "mock",
      "endpoint": "mock://localhost",
      "models": {
        "primary": "mock-model"
      },
      "enabled": true
    }
  },
  "prompt_templates": {
    "problem_analysis": "请分析以下运维问题并提供处置步骤:\\n\\n问题分类: {category}\\n问题描述: {description}\\n\\n请按照以下格式返回结构化的处置方案:\\n1. 问题分析\\n2. 处置步骤（按优先级排序）\\n3. 注意事项"
  }
}
EOF
```

#### 配置修正要点
- 确保 `active_provider` 对应一个 `enabled: true` 的提供商
- 若使用 OpenAI，需设置环境变量 `OPENAI_API_KEY`
- 可通过 `LLMConfigManager.createDefaultConfig()` 方法自动生成默认配置

### KNOWLEDGE_DIR_NOT_FOUND 错误
#### 含义说明
知识库目录未找到或为空，影响 `KnowledgeBaseService` 初始化。系统期望从 `knowledge-base/operation-procedures/` 和 `device-apis/` 加载 `.md` 文档作为智能决策依据。

#### 修复命令
```bash
# 创建知识库目录结构
mkdir -p knowledge-base/operation-procedures
mkdir -p knowledge-base/device-apis

# 添加示例运维文档
cat > knowledge-base/operation-procedures/cpu-high-usage.md << 'EOF'
# CPU 使用率过高处理

## 问题类型
性能问题

## 处置步骤
1. 使用 top 或 htop 查看占用最高的进程
2. 分析进程行为是否正常
3. 如为异常进程，终止并记录日志
4. 如为业务高峰，考虑扩容或优化

## 相关工具
- top
- htop
- ps aux
EOF

# 添加设备API文档
cat > knowledge-base/device-apis/server-monitoring-api.md << 'EOF'
# 服务器监控API

GET /api/v1/metrics/system
获取系统级指标（CPU、内存、磁盘）

GET /api/v1/status/service/{name}
查询指定服务运行状态

POST /api/v1/control/restart/{name}
重启指定服务
EOF
```

#### 配置修正要点
- 目录路径需与 `config/default.json` 中 `"knowledge.basePath"` 一致
- 支持 Markdown 和 API Blueprint 格式
- 推荐每类问题至少保留一份标准操作文档

### CONFIG_JSON_INVALID 错误
#### 含义说明
配置文件 JSON 格式错误，常见于引号缺失、逗号多余、中文字符编码等问题，将导致 Node.js 解析失败。

#### 修复方法
使用内置 JSON 验证命令：
```bash
# 验证所有配置文件语法
node -e "console.log(JSON.parse(require('fs').readFileSync('config/default.json', 'utf8')))" && echo "✓ 格式正确"
node -e "console.log(JSON.parse(require('fs').readFileSync('configs/llm-config.json', 'utf8')))" && echo "✓ 格式正确"
```

#### 推荐编辑器配置
- VS Code 安装 Prettier 插件自动格式化
- 开启 JSON Schema 校验以提示字段错误
- 避免手动输入引号和括号，使用自动补全

### PORT_IN_USE 错误（隐含检测）
#### 含义说明
虽然脚本未直接检测端口占用，但在后续启动阶段若发生冲突会导致服务无法绑定到指定端口（默认 3000）。

#### 修复命令
```bash
# 查看占用端口的进程
lsof -i :3000
# 或 Linux/macOS
netstat -an | grep 3000

# 终止占用进程（PID 替换为实际值）
kill -9 <PID>

# 修改配置更换端口
sed -i 's/"port": 3000/"port": 4000/g' config/default.json
```

#### 预防措施
- 在 CI/CD 流程中动态分配测试端口
- 使用 `process.env.PORT` 覆盖配置文件设置
- 启动前添加端口检查逻辑

## 自动化校验流程集成

### CI/CD 集成方式
建议将 `validate-system.sh` 作为 CI 流水线的预检步骤，典型 GitLab CI 配置如下：

```yaml
stages:
  - validate
  - build
  - test
  - deploy

system_validation:
  stage: validate
  script:
    - chmod +x validate-system.sh
    - ./validate-system.sh || { echo "系统验证失败"; exit 1; }
  artifacts:
    reports:
      dotenv: validation-report-*.txt
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
```

#### Jenkins Pipeline 示例
```groovy
pipeline {
    agent any
    stages {
        stage('System Validation') {
            steps {
                sh '''
                    chmod +x validate-system.sh
                    ./validate-system.sh
                '''
            }
            post {
                success {
                    archiveArtifacts 'validation-report-*.txt'
                }
            }
        }
    }
}
```

#### GitHub Actions 示例
```yaml
name: System Validation
on: [push, pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Run System Validation
        run: |
          chmod +x validate-system.sh
          ./validate-system.sh
      - name: Upload Report
        uses: actions/upload-artifact@v3
        if: always()
        with:
          path: validation-report-*.txt
```

## 建议的定期运行策略

### 运行频率建议
| 环境 | 建议频率 | 触发条件 |
|------|----------|----------|
| 开发环境 | 每次提交前 | pre-commit hook |
| 预发布环境 | 每日一次 | cron job (`0 2 * * *`) |
| 生产环境 | 每周一次 | 结合变更窗口 |

### 定时任务配置（Linux）
```bash
# 编辑 crontab
crontab -e

# 添加每日凌晨2点执行
0 2 * * * cd /path/to/AutoOperation && ./validate-system.sh >> logs/validation-cron.log 2>&1

# 添加每周一上午9点执行（生产环境）
0 9 * * 1 cd /opt/production/AutoOperation && ./validate-system.sh >> logs/prod-validation.log
```

### 报告归档与监控
- 自动生成带时间戳的报告文件：`validation-report-YYYYMMDD-HHMMSS.txt`
- 使用日志聚合工具（如 ELK）收集验证结果
- 设置异常报警规则（如连续三次失败触发企业微信通知）

### 最佳实践总结
1. **开发阶段**：启用 `simple-validate.sh` 作为 git hook，防止低级错误提交。
2. **CI 阶段**：运行 `validate-system.sh` 全面检查，阻断不合格构建。
3. **部署前**：再次执行深度校验，确保目标环境满足要求。
4. **运行期**：定期巡检，预防配置漂移和依赖丢失。

通过上述策略，可显著降低部署后故障率，提升系统稳定性与可维护性。
