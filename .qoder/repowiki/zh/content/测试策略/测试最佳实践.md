## 测试命名约定
清晰的命名是提高测试可读性的关键。测试用例应遵循“行为-状态-预期结果”的模式。

对于单元测试，采用描述性命名：
```javascript
test('应该成功创建会话', async () => { ... })
test('应该处理问题分析失败', async () => { ... })
```

对于集成测试，明确标识测试场景：
```javascript
describe('会话创建流程集成测试', () => { ... })
```

前端hook测试同样遵循此原则：
```typescript
test('应该成功创建会话', async () => { ... })
test('应该处理创建会话失败', async () => { ... })
```

## 测试粒度控制
合理的测试粒度确保每个测试用例只验证一个关注点，便于定位问题。

### 单元测试粒度
后端服务测试将功能分解为独立的`describe`块，每个块专注于特定功能模块：
```javascript
describe('会话创建', () => { ... })
describe('步骤执行', () => { ... })
describe('反馈处理', () => { ... })
```

每个`test`用例验证单一场景，如输入验证、正常流程、异常处理等。

### 前端Hook测试粒度
前端`useSession` hook测试同样按功能划分：
```typescript
describe('createSession', () => { ... })
describe('loadSession', () => { ... })
describe('executeStep', () => { ... })
```

这种细粒度的组织方式使得测试结构清晰，易于维护和扩展。

## 断言完整性
完整的断言确保测试能够全面验证系统行为，包括直接结果和间接影响。

### 后端服务断言
在`SessionManagementService`测试中，不仅验证返回值，还检查内部状态变化：
```javascript
// 验证返回结果
expect(result).toBeDefined();
expect(result.session.status).toBe('processing');

// 验证服务调用
expect(mockLLMService.analyzeProblem).toHaveBeenCalled();

// 验证状态更新
const updatedSession = sessionService.getSession(session.session_id);
expect(updatedSession.steps.length).toBe(2);
```

### 前端Hook断言
前端测试同样进行全面验证：
```typescript
// 验证API调用
expect(mockApiClient.createSession).toHaveBeenCalledWith({...});

// 验证状态管理
expect(setCurrentSession).toHaveBeenCalledWith(mockSession);

// 验证UI反馈
expect(toast.success).toHaveBeenCalledWith('会话创建成功');
```

这种多层次的断言策略确保了系统的各个层面都得到充分验证。

## 避免硬编码
避免在测试中使用硬编码值，提高测试的可维护性和可靠性。

### 使用动态生成的数据
测试中使用有意义的变量名而非魔法数字或字符串：
```javascript
const problemCategory = 'network';
const problemDescription = '网络连接超时';
const userId = 'user123';
```

### 配置共享测试数据
通过设置默认mock返回值避免重复定义：
```javascript
beforeEach(() => {
  mockLLMService.analyzeProblem.mockResolvedValue({
    category: 'network',
    severity: 'medium'
  });
});
```

### 使用工厂模式
对于复杂对象，可以考虑使用工厂函数生成测试数据，确保数据的一致性和可读性。
